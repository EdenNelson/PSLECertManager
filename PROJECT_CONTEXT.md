# Certificates Management Project - Context Document

## Project Overview

This PowerShell project automates SSL/TLS certificate for Windows Servers using Let's Encrypt via the Posh-ACME module. The system handles certificate creation, renewal, and deployment with secure secret management through BitWarden integration.

**Location:** `/Users/nelson/Documents/GitHub/PSLECertManager`  
**Author:** Eden Nelson  
**Project Type:** PowerShell Automation Scripts  
**Creation Date:** 2025  
**Version**: See [VERSION](VERSION) and [CHANGELOG.md](CHANGELOG.md)

---

## Project Purpose

The project provides an automated workflow for:

1. **Certificate Generation/Renewal** - Automated SSL/TLS certificate provisioning via Let's Encrypt (ACME)
2. **DNS Validation** - Integration with AWS Route53 for DNS-01 ACME challenge validation
3. **Secret Management** - Secure credential handling via BitWarden CLI
4. **Extensible Post-Processing** - Modular architecture supporting multiple post-action scripts in `PostScripts/` directory (ADFS, WAP, CMCMG)
5. **Scheduled Renewal** - Daily automated certificate renewal with configurable execution
6. **Dynamic Version Detection** - Auto-detects latest Posh-ACME version from local directory
7. **Log Rotation** - Automatic 1MB log cap with 5-file rotation and 90-day retention

**Post-Action Scripts**: Located in `PostScripts/` directory for organized deployment target management (Set-ADFSCert.ps1, Set-WAPCert.ps1, Set-CMCMGCert.ps1, Set-NPSCert.ps1).

## Project Architecture

### Core Scripts

#### 1. **Update-Certificate.ps1** (Main Orchestration)

- **Purpose:** Primary script for certificate creation and renewal
- **Key Functions:**
  - Loads variables from `Vars.psd1`
  - Imports secrets from `Secret.psd1` (generated by Invoke-SecretFile.ps1)
  - Imports Posh-ACME PowerShell module
  - Creates or renews certificates using Route53 DNS plugin
  - Executes post-processing scripts with certificate thumbprint
  - Runs under scheduled tasks for daily renewal

**Parameters:**

- `$PostScript` - (Required) Path to script executed after certificate operations. Can be provided via parameter or configured in `Vars.psd1`. Examples: `Set-ADFSCert.ps1`, `Set-IISCert.ps1`

**Workflow:**

1. Sets ACME server to Let's Encrypt Production (or Staging if `-UseStaging`)
2. Calls `Invoke-SecretFile.ps1` to refresh secrets from BitWarden
3. Loads configuration from `Vars.psd1` and imported secrets
4. Imports Posh-ACME module (auto-detects latest version)
5. Checks for existing certificates and renewal requirements
6. Creates new certificate or renews if expiring within 30 days
7. Executes post-script with certificate thumbprint and staging flag
8. Deletes secret files unless `-KeepSecrets` is specified

**Status:** Fully functional and tested

---

#### 2. **Invoke-SecretFile.ps1** (Secret Management)

- **Purpose:** Retrieves secrets from BitWarden and writes plaintext secret files
- **Key Functions:**
  - Validates BitWarden CLI tool presence
  - Imports BitWarden configuration from `BitWardenSecrets.psd1`
  - Uses BWS (BitWarden Secrets) token for authentication
  - Retrieves individual secrets via BWS CLI
  - Generates plaintext `Secret.psd1` file for use by other scripts

**Parameters:**

- `$DeleteBitWardenSecretsFile` - Switch to delete BitWarden config after secret retrieval
- `$DeleteSecretFile` - Switch to delete generated secret file
- `$BitWardenSecretsFilePath` - Path to BitWarden configuration (default: `BitWardenSecrets.psd1`)
- `$SecretFilePath` - Output path for secret file (default: `Secret.psd1`)

**Dependencies:**

- `bws.exe` - BitWarden Secrets CLI executable (expected in script root)
- `BitWardenSecrets.psd1` - BitWarden configuration file with token and secret IDs

#### 3. **Post-Action Scripts** (Certificate Deployment)

Located in `PostScripts/` directory. All scripts follow the same pattern:

- **Purpose:** Deploy certificates to specific services after successful certificate operations
- **Common Parameters:**
  - `$LatestCertThumbprint` - Certificate thumbprint to deploy
  - `$UseStaging` (optional) - Flag indicating staging vs. production
- **Called By:** `Update-Certificate.ps1` via `$PostScript` parameter

**Included Scripts:**

**Set-ADFSCert.ps1** (Active Directory Federation Services)

- Updates AD FS Service-Communications certificate
- Updates AD FS SSL certificate
- Restarts ADFS service to apply changes

**Set-WAPCert.ps1** (Web Application Proxy)

- Deploys certificate to WAP servers
- Updates WAP SSL bindings

**Set-CMCMGCert.ps1** (ConfigMgr Cloud Management Gateway)

- Deploys certificate to Azure CMG
- Updates ConfigMgr CMG configuration

**Set-NPSCert.ps1** (Network Policy Server)

- Validates certificate for NPS use
- Updates NPS registry configuration with new certificate thumbprint
- Restarts Network Policy Service (IAS) to apply changes

**Custom Post-Scripts:**

- Create additional scripts in `PostScripts/` for IIS, Exchange, or other deployment targets
- Follow the same parameter pattern for consistency

#### 4. **Register-CertificateScheduledTask.ps1** (Task Scheduler Integration)

- **Purpose:** Creates Windows Scheduled Task for daily certificate renewal
- **Execution Model:** Run manually by administrator with elevated privileges to create the scheduled task
- **Key Functions:**
  - Checks if scheduled task already exists
  - Creates new scheduled task if missing
  - Configures daily 2 AM execution with configurable random delay (default 30 minutes)
  - Configures task to run as SYSTEM account
  - Handles battery/power conditions gracefully

**Parameters:**

- `$MainDomain` - Primary domain for task naming
- `$PostScript` - (Required) Post-processing script to execute after renewal. Can be provided via parameter or configured in `Vars.psd1`
- `$RandomDelay` - (Optional) TimeSpan random delay for the daily trigger. Can be set as integer minutes in `Vars.psd1` (default 30)
- `$InitialDelay` - (Optional) TimeSpan delay for the one-time immediate trigger. Can be set as integer minutes in `Vars.psd1` (default 5)

**Task Details:**

- **Name:** `Renew-Certificates-{MainDomain}`
- **Path:** Cascade Technology Alliance
- **Triggers:**
  - One-time trigger: Executes once after specified delay (default 5 minutes)
  - Daily trigger: Runs at 2 AM with random delay (default ±30 minutes)
- **Principal:** NT AUTHORITY\SYSTEM (Highest privileges)
- **Start When Available:** Yes
- **Continue on Battery:** Yes

**Execution Context - SYSTEM Account:**

Once the scheduled task is created, ALL subsequent automated operations run as the SYSTEM account:

- `Update-Certificate.ps1` execution
- `Invoke-SecretFile.ps1` secret retrieval
- Posh-ACME certificate operations
- Post-action script execution (e.g., `Set-ADFSCert.ps1`)

**Posh-ACME Data Storage Location (SYSTEM Account):**

```text
C:\Windows\System32\config\systemprofile\AppData\Local\Posh-ACME\
```

This directory contains:

- ACME server configurations
- Account keys and metadata
- Certificate orders and authorizations
- Generated certificates (PEM, PFX files)
- Plugin data and state

**Important Testing Considerations:**

- Manual testing as a regular user creates certificates in: `C:\Users\{username}\AppData\Local\Posh-ACME\`
- Scheduled task execution uses SYSTEM's separate data location
- Certificate data **does not transfer** between user accounts
- For production-accurate testing, run as SYSTEM using PsExec: `PsExec.exe -s -i powershell.exe`
- Verify active data location: `Get-PAServer | Select-Object -ExpandProperty Folder`

### Configuration Files

#### 1. **Vars.psd1** (Variables Configuration)

```powershell
@{
    CertDomains       = "example.cascadetech.org"
    CertFriendlyName  = "SERVICE Let's Encrypt Certificate"
    PostScript        = "Set-ADFSCert.ps1"  # Optional: script to run after cert operations
    UseStaging        = $false               # Optional: use Let's Encrypt staging server
    KeepSecrets       = $false               # Optional: retain Secret.psd1 after execution
    RandomDelay       = 30                   # Optional: daily trigger random delay in minutes
    InitialDelay      = 5                    # Optional: one-time trigger delay in minutes
}
```

- Defines target domain(s) and certificate friendly name
- Can support multiple domains (comma-separated)
- **PostScript** (optional): Specifies which post-action script to execute after certificate creation/renewal
  - Examples: `Set-ADFSCert.ps1`, `Set-IISCert.ps1`, `Set-ExchangeCert.ps1`
  - Can be overridden via command-line parameter
- **UseStaging** (optional): When `$true`, uses Let's Encrypt staging server for testing (default: `$false`)
  - **Important:** Staging issues a **new certificate on every run** for testing purposes
  - Production mode reuses existing valid certificates
- **KeepSecrets** (optional): When `$true`, retains `Secret.psd1` after execution (default: `$false`)
- **RandomDelay** (optional): Random delay in minutes for the daily scheduled task trigger (default: 30)
- **InitialDelay** (optional): Delay in minutes before the one-time immediate task trigger executes (default: 5)
- Used for certificate creation and identification

#### 2. **BitWardenSecrets.psd1** (BitWarden Configuration)

```powershell
@{
    BitWardenSecrets = @{
        BWSToken = "<BITWARDEN_API_TOKEN>"  # API token for authentication
    }
    ScriptSecrets = @{
        Email        = "<SECRET_ID_EMAIL>"      # BitWarden secret ID
        R53AccessKey = "<SECRET_ID_ACCESS_KEY>" # BitWarden secret ID
        R53SecretKey = "<SECRET_ID_SECRET_KEY>" # BitWarden secret ID
    }
}
```

- **Purpose:** Maps BitWarden secret IDs for CLI retrieval (not the actual secrets)
- **Contains:** Access token for BitWarden Secrets Manager API
- **Security:** Contains credentials. Exclude from version control.
- **Lifecycle:** Deleted after `Invoke-SecretFile.ps1` execution (default) or retained with `-KeepSecrets` flag

#### 3. **Secret.psd1** (Generated Secrets - RUNTIME)

```powershell
@{
    Email        = 'first.last@cascadetech.org'
    R53AccessKey = '<AWS_ACCESS_KEY_ID>'
    R53SecretKey = '<AWS_SECRET_ACCESS_KEY>'
}
```

- **Purpose:** Runtime secrets imported by `Update-Certificate.ps1`
- **Generated By:** `Invoke-SecretFile.ps1` via BitWarden CLI
- **Contents:**
  - Email for ACME account contact
  - AWS Route53 access credentials
- **Security:** Contains plaintext AWS credentials. Excluded from version control.
- **Lifecycle:** Generated, imported into memory, then **deleted** after execution (secure default). Retained with `-KeepSecrets` for debugging.

### Configuration Priority

The scripts support flexible configuration through both command-line parameters and configuration files:

#### PostScript Configuration Priority

1. **Command-line parameter** (highest priority): `-PostScript "Set-ADFSCert.ps1"`
2. **Vars.psd1 configuration** (fallback): `PostScript = "Set-ADFSCert.ps1"`
3. **Error if missing**: PostScript is required; script will throw an error if not provided

**Implementation Details:**

- Both `Update-Certificate.ps1` and `Register-CertificateScheduledTask.ps1` check if the parameter was explicitly provided
- If not provided via parameter, scripts read `PostScript` value from `Vars.psd1`
- This allows different deployment targets (ADFS, IIS, Exchange) to use different post-action scripts
- Scheduled task creation can omit the `-PostScript` parameter when it's configured in `Vars.psd1`

**Usage Examples:**

```powershell
# Using parameter override (overrides Vars.psd1)
.\Update-Certificate.ps1 -PostScript "Set-IISCert.ps1"

# Using Vars.psd1 configuration (no parameter needed)
.\Update-Certificate.ps1

# Scheduled task creation with Vars.psd1 configuration
.\Register-CertificateScheduledTask.ps1 -MainDomain "example.com"
```

## Security Considerations

### Credentials Management

1. **BitWarden Integration:** Secrets stored in external vault, not in code
2. **Secret.psd1:** Runtime-generated file with plaintext credentials
   - **Risk:** Contains AWS credentials and email
   - **Mitigation:** Restrict file permissions, exclude from VCS
   - **Recommendation:** Consider encryption at rest

### Access Control

1. **Scheduled Task:** Runs as SYSTEM (highest privileges)
2. **File Permissions:** Ensure restricted access to configuration files
3. **BitWarden Token:** Sensitive - manage carefully

### Certificate Handling

1. **Let's Encrypt:** Production environment (not staging)
2. **Key Storage:** Managed by Posh-ACME in Windows certificate store
3. **ADFS Certificates:** Automatically deployed to trusted domain controller

## Configuration & Deployment

### Prerequisites

1. BitWarden account with Secrets feature enabled
2. BitWarden CLI (`bws.exe`) installed in project root
3. AWS Route53 access (credentials in BitWarden)
4. AD FS service running on target server
5. PowerShell 5.1 or higher
6. Local administrative privileges
7. (Optional) Sysinternals PsExec for SYSTEM account testing

### Initial Setup Steps

1. Configure BitWarden vault with required secrets:
   - Email address for ACME contact
   - AWS Route53 access key
   - AWS Route53 secret key

2. Create `BitWardenSecrets.psd1` with BitWarden token and secret IDs

3. Update `Vars.psd1` with target domain(s)

4. Place `bws.exe` in project root directory

5. Run `Invoke-SecretFile.ps1` to generate `Secret.psd1`

6. **Initial Testing (creates certificates in user's AppData):**

   ```powershell
   # Run as current administrator user for initial validation
   # PostScript will be read from Vars.psd1 if not specified
   .\Update-Certificate.ps1 -Verbose
   
   # Or override the Vars.psd1 setting:
   .\Update-Certificate.ps1 -PostScript "Set-ADFSCert.ps1" -Verbose
   ```

7. **Create scheduled task (requires administrative privileges):**

   ```powershell
   # Creates task that will run as SYSTEM
   # PostScript will be read from Vars.psd1 if not specified
   .\Register-CertificateScheduledTask.ps1 -MainDomain "example.cascadetech.org"
   
   # Or override the Vars.psd1 setting:
   .\Register-CertificateScheduledTask.ps1 -MainDomain "example.cascadetech.org" -PostScript "Set-ADFSCert.ps1"
   ```

8. **Production Testing as SYSTEM (optional but recommended):**

   ```powershell
   # Download PsExec from Sysinternals if not already available
   # Run PowerShell as SYSTEM to test production environment
   PsExec.exe -s -i powershell.exe
   
   # In the SYSTEM PowerShell session:
   cd C:\Scripts\Certificates  # or your installation path
   .\Update-Certificate.ps1 -Verbose  # PostScript read from Vars.psd1
   
   # Verify Posh-ACME data location
   Get-PAServer | Select-Object -ExpandProperty Folder
   # Should show: C:\Windows\System32\config\systemprofile\AppData\Local\Posh-ACME\acme-v02.api.letsencrypt.org
   ```

## Maintenance & Support

### Regular Tasks

- Monitor certificate renewal logs
- Verify ADFS service restarts successfully
- Check BitWarden secret access/rotation

### Troubleshooting

- Review PowerShell execution logs for errors
- Verify Route53 credentials haven't expired
- Check BitWarden token validity
- Confirm ADFS service is running
- **Certificate not found during scheduled execution:**
  - Verify Posh-ACME data location: `Get-PAServer | Select-Object Folder`
  - SYSTEM account location: `C:\Windows\System32\config\systemprofile\AppData\Local\Posh-ACME\`
  - User account location: `C:\Users\{username}\AppData\Local\Posh-ACME\`
  - Certificates created under one account are not visible to another account
  - Scheduled task runs as SYSTEM - ensure initial certificate creation happens as SYSTEM
- **Testing mismatch:** If manual testing works but scheduled task fails, verify execution account consistency
